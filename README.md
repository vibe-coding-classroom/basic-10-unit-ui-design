針對 **`basic-10-unit-ui-design` (網頁端模式控制介面設計：打造戰術面板)** 單元，這是學員將「後端邏輯」具象化為「專業工具」的終點站。它要求學員具備 **使用者體驗 (UX) 與安全設計** 的意識，學習如何透過視覺層級、觸覺回饋（震動）以及嚴謹的「權限隔離」機制，打造一個既直覺又能防止人為誤觸的工業級控制面板。

以下是在 **GitHub Classroom** 部署其作業 (Assignments) 的具體建議：

### 1. 範本倉庫 (Template Repo) 配置建議
UI 設計作業的核心在於「交互安全性」與「狀態可視化」，範本應包含前端框架與通訊介面，建議包含：
*   **📂 `src/PanelControls.js`**：**交互邏輯實作區**。學員在此實作按鈕點擊後的 CSS 類別切換，並調用 Web Vibrate API 實現物理報警。
*   **📂 `src/SyncEngine.js`**：**DOM 同步引擎**。學員在此實作監聽來自硬體的心跳報文，並強制將網頁 UI 修正為與實體一致的狀態。
*   **📂 `README.md`**：**UX 設計論證與測評**。提供表格讓學員記錄：為什麼選擇目前的配色方案？在自動駕駛模式下，如何確保搖桿不會被誤觸？
*   **📂 `docs/UI_STATES.md`**：**介面狀態對照表**。要求學員定義不同模式（Manual、Pilot、E-Stop）對應的視覺風格（光效、顏色、遮罩）。

### 2. 作業任務部署細節

#### 任務 1：戰術面板：CSS 狀態切換實作 (Tactical Panel & Authority Lab)
*   **目標**：透過視覺層級區分系統狀態，並實施「功能權限隔離」以提升物理安全性。
*   **Classroom 部署建議**：
    *   **三模分段選單**：設計具備高對比高亮效果的模式切換鈕。
    *   **控制權隔離**：**核心關鍵點**。當系統切換至 `Pilot` (自動駕駛) 時，手動遙控區（搖桿、微調鍵）必須變為灰色背景且 `pointer-events: none` 鎖死交互。
    *   **驗證要點**：視覺上是否一眼就能看出目前的模式？自動駕駛模式下，是否真的無法透過網頁遙控干擾機器人？

#### 任務 2：雙向狀態同步與校正 (Bi-directional State Sync Lab)
*   **目標**：打破「樂觀 UI」的假象。確保面板反映的是機器人的「物理現實」，而非使用者的「點擊紀錄」。
*   **Classroom 部署建議**：
    *   **真值追蹤器**：實作每 200ms 解析一次硬體回傳的 `current_mode`。
    *   **強制一致性**：若用戶點了「自動」，但硬體因電力不足跳回「手動」，網頁按鈕必須在收到心跳後立即自動彈回「手動」位置。
    *   **驗證要點**：重啟機器人後，網頁面板是否能自動恢復到與機器人一致的顯示狀態？「眼見是否即真實」？

#### 任務 3：「E-Stop」緊急停機與警報 (Haptic & Visual Alert Lab)
*   **目標**：打造最高優先權的安全屏障。透過多維度（視覺、觸覺）的反饋告知使用者緊急狀態，並設計預防性的解鎖流程。
*   **Classroom 部署建議**：
    *   **全域緊急遮罩**：按下 E-Stop 後，畫面上方出現滿版紅色半透明遮罩，封鎖所有正常的控制路徑。
    *   **震動報警實作**：調用 `navigator.vibrate([200, 100, 200])` 產生緊急物理脈衝。
    *   **安全長按解鎖**：**技術細節**。嚴禁單擊解除。必須實作「長按 2 秒」才能撤銷 E-Stop 狀態。
    *   **驗證要點**：遮罩是否優先於所有 UI 元件？震動回饋是否顯著提升了警報感知？長按機制是否能有效防止誤觸？

### 3. 介面設計與交互安全性點評標準 (UI Design & Safety Standards)
此單元的價值在於 **「對使用者心智模型的理解與對物理安全的操作承諾」**：
*   [ ] **視覺層級清晰度**：緊急按鈕是否足夠顯眼？當前選中模式是否具備壓倒性的辨識度？
*   [ ] **交互防誤觸機制**：是否在邏輯層面（而非僅是視覺）實施了權限鎖定？
*   [ ] **通訊同步透明度**：面板數據是否始終與硬體真值保持一致，不產生短暫的「認知不對稱」？
